FreelanceHub — Function-by-Function Reference

A complete walkthrough of every file and function in the project: what it takes, what it returns, and why it exists. FreelanceHub is a role-based freelance job-board built on Next.js 16, React 19, Prisma + PostgreSQL, Better Auth, and Zod.

How to read this: the document is grouped by layer — Data layer Auth & helpers API routes Pages Components. Each function appears as a dark signature bar followed by an explanation of its Inputs, Returns, and Logic.

1. Data & Infrastructure Layer

lib/db.ts — the Prisma client singleton

Creates one shared database connection for the whole app and caches it during development.

prisma : PrismaClient  // exported singleton

Why: Next.js hot-reloads modules constantly in dev. Without caching, each reload would spawn a brand-new PrismaClient and exhaust the database's connection pool.

Logic: it looks for an existing client on globalThis.prisma; if none exists it creates one (?? new PrismaClient()). In any non-production environment it stores the client back on globalThis so the next reload reuses it. In production a fresh single instance is used.

export const prisma = globalForPrisma.prisma ?? new PrismaClient()
if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma
prisma/schema.prisma — the data model

Defines the tables and relationships. Five models plus an enum drive everything.

ModelPurpose & key fields
UserRole (enum)Either CLIENT or FREELANCER — the backbone of all permissions.
UserAccount profile: email (unique), role, bio. Relations to sessions, accounts, jobs and applications.
Session / Account / VerificationTables required by Better Auth to store login sessions, credential records (incl. hashed password), and email tokens.
Jobtitle, description, budget (Int), and clientId → the owning User. Has many applications.
ApplicationLinks a freelancerId to a jobId with a coverLetter. @@unique([freelancerId, jobId]) enforces one application per job per freelancer at the database level.

2. Authentication Setup

lib/auth.ts — server-side Better Auth instance
auth = betterAuth({ … })

Logic: wires Better Auth to the database through the prismaAdapter, enables the emailAndPassword strategy, and installs the nextCookies() plugin so session cookies are set automatically in Next.js responses.

Key detail — custom fields: role (required) and bio (optional) are declared under user.additionalFields. This is what lets a plain auth library carry the app's CLIENT/FREELANCER concept inside the session object.

Secrets: reads BETTER_AUTH_SECRET and BETTER_AUTH_URL from the environment.

lib/auth-client.ts — browser-side auth client
authClient = createAuthClient({ … })

Returns: a client exposing signIn.email(), signUp.email(), signOut(), and the useSession() React hook.

Key detail: inferAdditionalFields mirrors the server's role field so the client-side session is fully typed and session.user.role is available in components.

app/api/auth/[...all]/route.ts — auth endpoint
export const { GET, POST } = toNextJsHandler(auth)

A catch-all route ([...all]) that forwards every auth request (sign-in, sign-up, sign-out, session) to Better Auth in a single line. No custom logic needed.

3. Session & Permission Guards

Two parallel files provide the same guards in two contexts: session.ts for Server Components/pages (reads cookies via headers()) and session-request.ts for API route handlers (reads from the passed request). Each guard returns a discriminated union — either { error: Response } or a success object — so callers branch with "error" in result.

lib/server/session.ts (page context)
async getSession() : Promise<Session | null>

Logic: calls auth.api.getSession({ headers: await headers() }) to read the current user from the request cookies. Returns null if nobody is logged in.

async requireSession() : { error } | { session }

Logic: calls getSession(); if there is no session.user it returns { error: 401 "You must be logged in" }. Otherwise returns { session }.

async requireRole(allowedRoles: ("CLIENT"|"FREELANCER")[])

Inputs: a list of roles permitted to proceed.

Logic: first runs requireSession() (bubbling up any 401). Then checks session.user.role against the allowed list; if not included, returns { error: 403 }. Otherwise passes the session through.

async requireJobOwner(jobId: string) : { error } | { session, job }

Logic: requires a session, then loads the job via getJobById(jobId). Returns 404 if the job is missing, 403 if job.clientId !== session.user.id. On success it conveniently returns both the session and the fetched job so the caller needn't query again.

lib/server/session-request.ts (API context)

Mirror images of the above — getSessionFromRequest, requireSessionFromRequest, requireRoleFromRequest, requireJobOwnerFromRequest — but each takes the NextRequest and reads request.headers instead of the global headers(). Behaviour and return shapes are identical.

Why two copies? Server Components can use the ambient headers() API, but API route handlers receive their own request object. Splitting them keeps each context using the correct cookie source.

4. Validation Schemas & Error Formatting

lib/validations/auth.ts
SchemaRules
loginSchemaemail must be valid; password required (min 1).
registerSchemaname ≥ 2 chars, valid email, password ≥ 8 chars, role ∈ {FREELANCER, CLIENT}.

LoginFormValues and RegisterFormValues are derived with z.infer, giving forms their TypeScript types automatically.

lib/validations/job.schema.ts

createJobSchema: title ≥ 5 chars, description ≥ 20 chars, budget a number ≥ 50. Reused for both create and update.

lib/validations/application-schema.ts

applyJobSchema: coverLetter between 20 and 2000 characters.

lib/format-api-error.ts
formatApiError(error: unknown) : string

Purpose: turn the various error shapes an API may return into a single human-readable string for an alert().

Logic: if the error is already a string, return it. If it's a flattened Zod error object, collect all fieldErrors values (and fall back to formErrors), joining them with newlines. Otherwise return a generic "Something went wrong".

5. Service Layer (all Prisma queries live here)

services/jobs.service.ts
async createJob(data: { title, description, budget, clientId }) : Job

Inserts a new job row owned by clientId and returns the created record.

async getJobs() : Job[]

Returns every job, include-ing its client (poster), newest first (orderBy createdAt desc). Powers the jobs list page.

async getJobById(jobId: string) : Job | null

Fetches a single job with its client and all applications. Used by the detail page and the ownership guard. Returns null if not found.

async updateJob(jobId, data: { title?, description?, budget? }) : Job

Updates the given fields of a job (all optional) and returns the updated row.

async deleteJob(jobId: string) : Job

Deletes the job by id.

async getJobOwnerId(jobId: string) : string | null

A lightweight lookup that selects only clientId — useful for ownership checks without loading the whole job. Returns null if the job doesn't exist.

services/applications.service.ts
async createApplication(data: { jobId, freelancerId, coverLetter }) : Application

Creates an application and returns it with the applicant's public freelancer fields (id, name, email, role) included.

async getApplicationsByJobId(jobId: string) : Application[]

Lists all applications for a job, newest first, each with the freelancer's name & email. Shown to the job owner.

async hasApplied(jobId, freelancerId) : boolean

Looks up the unique (freelancerId, jobId) pair and returns Boolean(application). Prevents duplicate applications and decides which UI to show.

async getApplicationByFreelancerAndJob(jobId, freelancerId) : Application | null

Returns the actual application record for that pair (or null) — the data-bearing counterpart to hasApplied.

6. API Route Handlers

app/api/jobs/route.ts
export async function GET() : Response

Public. Calls getJobs() and returns the array as JSON. On failure logs and returns 500.

export async function POST(request) : Response

Create a job — CLIENT only. Step by step:

  • requireRoleFromRequest(request, ["CLIENT"]) → returns its error (401/403) if not a client.
  • Parse JSON body; coerce budget to a Number; createJobSchema.safeParse(…).
  • On validation failure → 400 with the flattened Zod error.
  • Call createJob({ ...data, clientId: session.user.id }) — the owner is taken from the session, never trusted from the body.
  • Return the new job with 201.
app/api/jobs/[jobId]/route.ts
export async function GET(request, { params }) : Response

Awaits params to get jobId, fetches the job; returns 404 if missing, else the job JSON.

export async function PUT(request, { params }) : Response

Update — owner only. Runs requireJobOwnerFromRequest(request, jobId), re-validates the body with createJobSchema, then calls updateJob(). Returns the updated job or the appropriate error code.

export async function DELETE(request, { params }) : Response

Delete — owner only. Ownership guard, then deleteJob(), returning { success: true }.

app/api/jobs/[jobId]/apply/route.ts
export async function POST(request, { params }) : Response

Apply to a job — FREELANCER only. A chain of guards, each returning a specific status:

  • requireRoleFromRequest(["FREELANCER"]) → 401/403.
  • Job must exist → else 404.
  • Cannot be your own job (job.clientId === session.user.id) → 403.
  • Cannot apply twice (hasApplied) → 409 Conflict.
  • applyJobSchema.safeParse(body) → 400 on bad input.
  • Create the application and return it with 201.

7. Middleware

middleware.ts
export function middleware(request: NextRequest) : NextResponse

Runs before matched routes render. Reads the session cookie with getSessionCookie(request); if absent, redirects to /login, otherwise calls NextResponse.next() to continue.

config.matcher limits it to private areas: /dashboard/*, /jobs/create, /jobs/:jobId/edit, /messages/*, /profile/*.

This is only a fast first gate based on cookie presence — the API routes and pages still re-verify the real session and ownership, so security never depends on this check alone.

8. Pages (App Router)

app/layout.tsx — RootLayout

Server Component wrapping every page. Loads the Geist / Geist_Mono fonts via next/font, sets global metadata, and renders the persistent <Navbar/>, the page {children}, and a footer with the current year.

app/page.tsx — Home (Client Component)

Demonstrates client-side state: a useEffect + setTimeout shows the <Loading/> skeleton for 500 ms, then renders two columns — Featured Jobs (filtered by job.featured) and All Jobs — from a hard-coded demo array, each rendered through <JobCard/>.

app/jobs/page.tsx — JobsPage (Server Component)

async component that calls getJobs() directly (no API round-trip), shows an empty state, or lists each job as a <Link> to its detail page. Renders <CreateJobLink/> in the header.

app/jobs/[jobId]/page.tsx — JobPage (Server Component)

The richest page. Awaits params, loads the job (notFound() if missing), then reads the session and computes flags: isOwner, isFreelancer, isClient. Based on these it conditionally renders:

  • Owner → <JobOwnerActions/> + the applicant list (loaded via getApplicationsByJobId).
  • Freelancer → <ApplyJobForm/>, or a "already applied" message (decided by hasApplied).
  • Guest → a login prompt. Other client → "only freelancers can apply".
app/jobs/create/page.tsx — CreateJobPage (Client Component)
async function createJob(data: JobFormData)

POSTs the form to /api/jobs with credentials. On error, alerts a formatted message; on success, redirects to /jobs. A useEffect guard redirects non-clients/guests away, and the page renders null until the session is confirmed.

app/jobs/[jobId]/edit/page.tsx — EditPage (Server Component)

Server-side gate: loads job + session, calls notFound() if missing, redirect("/login") if logged out, and redirect("/jobs/[id]") if not the owner. Then renders <EditJobForm/> pre-filled with the job's current values.

app/login/page.tsx & app/register/page.tsx (Client Components)
async function onSubmit(data)  // login

Validates via useForm + zodResolver(loginSchema), calls authClient.signIn.email(), then router.refresh() and redirects by role (CLIENT → /jobs/create, else /jobs). Shows an alert on bad credentials.

async function onSubmit(data)  // register

Validates with registerSchema, calls authClient.signUp.email() (passing the custom role), then redirects to /login on success.

app/skills/page.tsx — SkillsPage (Server Component)

Demonstrates server-side data fetching from an external API: await fetch("jsonplaceholder…/users"), then maps results into <UserCard/> components.

app/loading.tsx — Loading

An App-Router special file shown automatically during navigation/suspense. Renders an animated animate-pulse skeleton mimicking the home layout.

9. Components

components/Navbar.tsx (Client)

Uses authClient.useSession() to render adaptively: brand link + "Skills"/"Jobs" always; "Post a Job" only for CLIENT; when logged in shows the user's name, role and a signOut() button; otherwise Log in / Register buttons.

components/JobForm.tsx (Client) — shared create/edit form
JobForm({ defaultValues?, onSubmit, buttonText })

A reusable form wired with useForm + zodResolver(createJobSchema). Renders title/description/budget inputs with inline error messages and a submit button that shows "Saving…" while isSubmitting. The parent supplies the onSubmit behaviour — so the same component powers both creation and editing.

components/EditJobForm.tsx (Client)
async function updateJob(data: JobFormData)

Wraps JobForm with defaultValues from the existing job. PUTs to /api/jobs/[jobId]; on success redirects to the job detail page, on error alerts the formatted message.

components/ApplyJobForm.tsx (Client)
async function onSubmit(data: ApplyJobFormValues)

Cover-letter form validated by applyJobSchema. POSTs to /api/jobs/[jobId]/apply; on success calls router.refresh() (re-rendering the server page into the "already applied" state) and alerts success.

components/JobOwnerActions.tsx (Client)
async function handleDelete()

Shows a confirm() dialog, then DELETEs /api/jobs/[jobId], toggling a deleting state for the button label. On success navigates to /jobs and refreshes. Also renders an "Edit Job" link.

components/CreateJobLink.tsx (Client)

Reads the session and renders the "Create Job" link only if the user's role is CLIENT — returns null otherwise (and while the session is pending).

components/JobCard.tsx (Client)
function handleApply()

Demo card with local applied state: clicking "Apply Now" sets applied=true, switching the button to a disabled "Applied". Displays title, company, and budget in ETB. (Used on the demo home page.)

components/UserCard.tsx (Server-friendly)

Pure presentational component rendering a user's name and company.name — consumed by the Skills page. Exports a UserCardProps type.

10. End-to-End Recap

Post a job: JobForm → POST /api/jobs → requireRole(CLIENT) → safeParse → createJob() → Prisma → 201 → /jobs

Apply: ApplyJobForm → POST /api/jobs/[id]/apply → requireRole(FREELANCER) → exists? own? duplicate? → safeParse → createApplication() → 201 → refresh

View applicants: JobPage (server) → getSession → isOwner → getApplicationsByJobId() → render list
Design theme: every mutation passes the same gauntlet — authenticate → authorize (role/ownership) → validate (Zod) → service → database — and that gauntlet is repeated at the middleware, page, and API levels. Reads happen directly in Server Components for speed; writes go through typed REST routes.
